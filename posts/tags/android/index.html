Published Docker image of up-to-date XashDS​

by SNMetamorph​

If you have some VDS or PC with installed Linux distro, now it's possible to setup Half-Life dedicated server based on up-to-date Xash3D FWGS engine build in couple of actions! Also, this Docker image have pre-installed Metamod, AMX Mod X, and bots using jk_botti Metamod plugin.

For now, pre-built Docker image only available for Half-Life, but in future we hope to build images for other mods too (such as Counter-Strike 1.6).

Here is Docker Compose file example. You can read more information at GitHub repository.

services:
  xashds:
    image: snmetamorph/xashds-hldm:latest
    container_name: xashds-hldm
    restart: always
    command: +map crossfire
    ports:
      - '27015:27015/udp'

Fixed broken physics behavior on high FPS​

by SNMetamorph and a1batross​

This bug was caused by rounding error in physics code, because of it player movement physics was working weird with 200+ FPS values.

Framerate limitation changes​

by a1batross​

Soft limit for framerate is 200 FPS. Hard limit is 1000 FPS - and you can enable it with fps_override console variable, but also make sure that gl_vsync is disabled.

GoldSrc network protocol support​

by a1batross​

For now, GoldSrc protocol support is in mainline engine and available to testing for anybody. This feature is still work-in-progress, but anyway it is good-enough to be called playable. We appreciate any bug-reports and contributions around it.

For connecting to GoldSrc-based servers, you can use console command: connect <ip> gs. Also, GoldSrc servers can be added to favorites list - just don't forget to choose proper protocol when adding it.

There is short footage of playing Counter-Strike 1.6 on some GoldSrc-based server.

But keep in mind, there are requirement for server to be able to accept connections from Xash3D-based clients: it should use Reunion or Dproto. Without this requirement, you will just get "Steam validation rejected" error on connecting. That is because proper authorization with Steam API is not implemented in engine yet. There is list of things which still unfinished:

  • Authorization using Steam API (first priority)
  • Voice chat on GoldSrc-based servers
  • Player sprays on GoldSrc-based servers

Also, we encountered that some GoldSrc-based servers are recognizing Xash3D clients as "fake clients" and banning/kicking them. Maybe this problem will be solved along with better compatibility with GoldSrc behavior, but may be not - we don't know logic behind this useless "fake client" checks.

Compatibility with GoldSrc mods improved further​

by Flying With Gauss​

There isn't much to describe here, as it involves a plenty of minor changes that are hard to track down retrospectively. In some cases, incompatibility caused by difference in console variables. Sometimes, it was caused by difference in internal logic between GoldSrc and Xash3D.

But, some of the facts: Sven Co-op 4.8 client started working on Xash3D FWGS.

Added SIGTERM signal handling for XashDS​

by SNMetamorph​

This feature needed for graceful shutdown of dedicated server inside Docker container, and in case of using XashDS as systemd service.

Enabled engine binaries signing for Windows platform​

by SNMetamorph and a1batross​

We were annoyed with a lot of user reports about engine being false-positively detected by antiviruses as malware, especially by Microsoft Defender. So, we introduced engine binaries signing - and it seemed to work, even despite the fact that we're using self-signed certificate for this procedure. We have not received any false-positive detection reports anymore.

Complete IPv6 support in multiplayer​

by Flying With Gauss​

At this day, engine and masterserver both supports IPv6. And we have some masterservers running on the Internet through IPv6. This means anybody could setup their Xash3D FWGS server on the Internet with IPv6 and it will shown for other clients who also have IPv6 address. Also, it's well tested and we didn't discovered any issues with it.

If you for some reason need to run dedicated server exclusively on IPv6, you can use -noip startup parameter, which completely disables IPv4 networking.

Voice chat improvements​

by SNMetamorph and a1batross​

For now, it is almost perfect working, even in case when several players are talking simultaneously. Also, some settings related to voice chat were added to multiplayer settings menu. Voice chat button was added to touch configuration for Android port.

There is short video with demonstration of working voice chat in multiplayer.

Player sprays improvements​

by SNMetamorph and a1batross​

Added console variable cl_logomaxdim for setting maximal dimension for player sprays. By default it is 96 units.

Logging improvements​

by SNMetamorph and a1batross​

Added more information to logs, such as: engine build number, VCS branch name & commit hash, platform and architecture name, game startup parameters. This was made to provide better experience of processing users feedback about bugs or their other issues with engine/mods.

Sounds overriding through sounds.lst​

by a1batross​

Using sounds.lst located in scripts folder, modder can override some of the hardcoded sounds in temp entities and server physics. You can read more detailed description in documentation.

Added gl_overbright from early GoldSrc versions​

by a1batross​

Also, this includes some fixes according to gamma that were made to match GoldSrc looking.

Server browser improvements​

by Flying With Gauss​

Added tabs with favorites and recent servers, also basic servers filtering was added (in the bottom).

Also, entirely new menu for viewing server detailed information was added.

PHS generating​

by a1batross​

This is the same as PAS, id Software called it Potentially Hearable Set, Valve calls it Potentially Audible Set. Pre-calculated PHS is present only in Quake 2 maps, in QuakeWorld and Half-Life it is calculated from PVS. The calculation is relatively fast, but with the increase in the number of leaves in BSP it eats up memory exponentially (at least in compressed form).

This feature helped with fixing issue #619 and providing better compatibility with GoldSrc behavior.

Support for HD textures​

by a1batross​

This feature allows to create HD textures packs for classic mods, without involving any code changes. Initially, this feature was presented in vanilla Xash3D, but was removed during separating renderer and engine to different libraries back in 2019. So, now it's back here again. You can read more detailed description in documentation.

Support for Ogg Vorbis and Ogg Opus audio formats​

by SNMetamorph​

We've needed some kind of MP3 alternative for a long time. Since being invented back in 1993, nowadays MP3 is far inferior to its modern alternatives in both compression efficiency and audio quality. At same time, also the best lossy audio compression codec Opus are being open-source and free, and we're already using it for voice chat - why don't use it for audio files too? Also, some of the mods using really a lot of sounds, and at some point audio compression efficiency can save several gigabytes of disk space.

What about Vorbis codec: it is still better than MP3, but from practical point of view it can be useful in terms of backward compatibility with existing content (because, obviously, if you have lossy-compressed audio - it's better to not recompress it to another format, because this will lead to "generation loss"). Ogg Vorbis is pretty wide-spread audio format, also other idTech-based game engines supports it.

Sound looping feature, that already supported in MP3, now supported also in both these formats too.

tip

If you not yet familiar: for Ogg Vorbis sounds file extension .ogg is used. For Ogg Opus sounds used .opus file extension respectively.

FastDL improvements​

by a1batross​

Implemented support for chunked transfer encoding and compression with gzip in HTTP client. Also, there is HTTPS support in plans, but we don't have any ETA for this.

Video playback support​

by a1batross​

This is from that rare kind of features, that already were in engine for a long time, but could not be used for technical reasons. Originally, video playback was dependent on platform-specific Video For Windows framework and ancient Cinepak codec: for proper working, it should be installed system-wide and also it's available for use only for 32-bit processes and exclusively for Windows. Because of that, users were having a lot of troubles while trying to get things working, and with some curcumstances it was not possible at all.

In engine and some of mods, we support (and use) 64-bit builds for a long time already. Also, we do care about other platforms rather than only Windows. This is why we decided to completely get rid of VFW-related code, and instead use ffmpeg for video playback. Also, we decided that video playback support should be an optional feature - ffmpeg libraries are bulky, also most of mods doesn't uses videos anyway. So, anybody who have plans to distribute mod along with engine, or just free up some of disk space, will be able to just remove that libraries.

Here is short demonstation of video playback feature within PrimeXT. Original valve.avi intro video was used for it.

Android port and launcher improvements​

by a1batross and Velaron​

Launcher was improved, some Android-specific bugs were fixed too.

Reduced connection time to the servers​

by SNMetamorph​

This feature appeared as a result of changing pre-connection bandwidth test logic: test stages duration and count were reduced. For now, you can connect to server in less of 5 seconds (except time spent on content downloading, if it was required)

MacOS port rehabilitated​

by sofakng​

Project building issues on MacOS were resolved, CI auto-builds for MacOS were added. But it's not well-tested anyway, there is possibility to catch some MacOS-specific bugs, in this case feel free to report about it on GitHub. But our capabilities with MacOS port are limited, since nobody from developers have any relevant hardware, and we're looking for proper maintainer because of it.

Port for Sega Dreamcast (WIP)​

by maximqaxd​

This is still in progress, but for now it already works on actual hardware. Port can load content that was originally made for Dreamcast port, because it supports all of that platform-specific content formats such as PVR with VQ compression and ADPCM WAV sounds. Multiplayer works too.

Compiling Half-Life 1 maps on Android TV-box

January 20, 2023 · 8 min read

I have been modding the legendary Half-Life 1 for these years, and somehow quite spontaneously I thought: how cool it would be to be able to compile maps on some platform other than the usual x86 and amd64... Then I remembered that I have an Android TV box at home that, in theory, could be used for this purpose. So, why not? This is how all this things was started.

What's we have?​

In stock, I had a budget GoldMaster I-905 TV set-top box, which was completely unremarkable. It has 16 gigabytes of internal memory, 2 gigabytes of RAM. Based on the Allwinner H313 processor. From interfaces: has two USB 2.0 ports, composite video output, HDMI port. Network access can be provided both via Wi-Fi and via an Ethernet port (I used a wired connection).

Main information about TV-box system, shown by neofetch

Also, we need map compiler sources. For almost 30 years of the existence of the game, quite a few of their variations have appeared: the original compilers from Valve, ZHLT, VHLT, P2ST, etc. I will use the map compilers from my PrimeXT project: they are the most up-to-date at the moment, they will be the easiest to assemble and run under some exotic platform, unlike all other variations. The game itself, as well as all related utilities, including map compilers, are written in C/C++. PrimeXT uses the CMake build system.

Preparing​

After a short search for information, I found out that in order to build C++ projects straight on Android, I need Termux. It needs to be downloaded from the F-Droid repository as the version published on Google Play has been deprecated and is no longer supported.

After installing Termux, just in case, update the installed packages:

pkg upgrade

Now let's install the packages that we actually need.

pkg install cmake git openssh pkg-config ninja

For more convenient work, I decided to run an SSH server on the TV-box and connect to it from my computer. To do this, I used a bunch of software KiTTY + WinSCP. Of course, you can use any other software instead. And you can do without an SSH server at all - just write all the necessary commands into Termux manually on the device. I will use WinSCP for convenient work with files on a remote device.

Now you need to start the SSH server: the procedure is quite trivial.
First, we need to set a password to connect to the server, and then start the corresponding service.

passwd
sshd

After that, you can try to connect from the computer to the server. Please note that Termux uses port 8022 for the SSH server, not the usual 22. You can enter any username, but you need to enter the password specified earlier. If the connection is successful, a shell will be displayed.

Building map compilers​

After completing all the preparatory procedures, you can go directly to the build procedure: the first step is to clone the sources into your home directory.

cd ~
git clone --recursive https://github.com/SNMetamorph/PrimeXT.git
cd PrimeXT

Next, you need to bootstrap vcpkg package manager. You can be warned about missing packages, so you'll need to install they too.

export VCPKG_FORCE_SYSTEM_BINARIES=1
external/vcpkg/bootstrap-vcpkg.sh

Then, you need to create and prepare the directory in which the assembly will take place.

cmake -E make_directory ./build
cd build

And now it's time to generate the CMake cache and run the build. Since I'm only interested in map compilers, I disabled building game libraries, the launcher, and other utilities. Also, to make the compilation of the project faster, I will set the build to 4 threads.

cmake .. --preset utils-termux-debug
cmake --build . --parallel 4
CMake cache was successfully generated

After executing the two commands described above, the build process will start.

If the build succeeds, it will look like this. If any errors occur, it is better to restart the assembly in one thread, so it will be easier to find out exactly where the compilation error occurred. By the way, on this set-top box, the assembly into 4 streams took about four minutes, taking into account the fact that an IPTV player was running in the background.

Map compilers build successfully finished

Compiling map​

For the final step, you need to prepare a working directory that will contain WAD files with textures, a .rad file, and a map source. In my case, this directory will be called gameres.

mkdir ~/gameres
cd ~/gameres

Then, inside this directory, you need to create a gameinfo.txt file with the following content:

basedir ""
gamedir "gameres"

Important tip: in this file, the value of the gamedir parameter must match the name of the working directory. Next, you need to put the WAD files used in the working directory, create a subdirectory called maps inside it, and put the .rad file and the map source in the form of a .map file into the subdirectory. All the necessary files that I used are available at this link.

Now you can try to compile the map, and all that remains is to wait.

cd ~/PrimeXT/build/Release/primext/devkit
./pxcsg ~/gameres/maps/crossfire_double1_v2.map
./pxbsp ~/gameres/maps/crossfire_double1_v2.bsp
./pxvis ~/gameres/maps/crossfire_double1_v2.bsp
./pxrad ~/gameres/maps/crossfire_double1_v2.bsp

The map compilation process consists of four steps. At the end of each stage, the time spent on it will be displayed. If there are any problems during the compilation of the map, you can look at the log file in the maps subdirectory. Also, you can add the -dev 5 startup option to compilers to display the process in more detail.

Last map compiling stage was finished

Checking the results​

After the map successfully compiled on the TV-box, I decided to rebuild the same map on my computer and then compare these two maps. Ideally, there should be no visible differences. I will test this whole thing in the PrimeXT mod, which is based on the Xash3D FWGS engine.

Maps build on different architecture and platform

Personally, I didn’t notice any visual differences in lighting or geometry at all, which of course is good. The experiment can be considered successful.

Resume​

Using the procedures described above, it is quite possible to compile maps for Half-Life 1 on almost any device that is based on the Android or Linux OS. Unless, on too large and detailed maps, the device may not have enough RAM for compilation. Can all this be useful to someone? Of course. For example, if for some reason you don’t want to load the main PC, or there are no devices other than a smartphone at all, but the map needs to be compiled somehow.
